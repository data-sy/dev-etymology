검색 화면 — 헤더 위계 · 수직 리듬 · 탭바 간격

A안(하단 검색필드 + 하단 탭바) 확정 위에서, 보고된 3개 문제만 손본 변형.
토글로 평상시 / 입력중, 기기폭(Pro / 13 mini)을 비교.

각 안이 3개 문제를 다루는 법

① 헤더 위계/위치 ② 탭바↔필드 간격 ③ 수직 리듬
B · 좌상단 앵커 (추천) 헤더를 전체폭 좌측 상단에 고정(센터링 부작용 제거). 크기는 토큰 유지(serif 20). 위에 safe-area + 8 여백을 줘 "코너에 박힌 로고"로 읽히게. field 아래 14pt + 탭바 자체 top padding 으로 명확히 분리. mini에서도 숨 쉼. header→body 20, hint→field 10, field→tabbar 14. 절제된 증가.
C · 헤더 승격(hero) 헤더를 serif 28(titleHero)로 키우고 좌상단 고정. "이게 앱 정체성"이 단박에 읽힘. 부제는 그대로 mono. B와 동일 14pt 분리. hero가 커진 만큼 header→body 24로 더 넉넉. 시원하지만 빈 화면이 더 비어 보일 위험.
D · 헤더+힌트 통합 카드 좌상단 헤더(B와 동일). 추가로 힌트를 필드 placeholder에 흡수, 필드 위 안내문 제거 → 바닥 크롬 1줄 감소. 힌트 줄이 사라져 field→tabbar 16pt까지 여유. mini 최적. 바닥이 가장 가벼움. 단 "왜 영문?" 맥락이 약해짐(placeholder만으로).

추천: B · 좌상단 앵커

심플 우선 + 토큰 유지 원칙에 가장 충실. 보고된 3문제의 원인(센터링 부작용·과밀 패딩)만 정확히 제거하고 장식을 안 늘린다. C(hero)는 헤더가 커지며 빈 본문이 더 휑해 보이고, D는 힌트를 빼 "영문 입력" 맥락이 약해진다(한글 사용자에게 영문 키워드 앱이라는 안내는 가치 큼). mini·Pro 양쪽에서 검증됨.

접근성 메모 (전 안 공통)

· 헤더는 accessibilityElement(.combine) 유지 → VoiceOver가 "DevEtym 개발 용어 어원 사전" 한 덩이로 읽음(현행 유지).
· 힌트 textMuted(#8A8A8A) on bg(#0A0A0A) 대비는 보조 안내 수준에선 허용. 단 본문 격상은 금지(작은 한글 본문엔 textDim 이상).
· Dynamic Type 확대 시 본문(최근검색/empty)이 커지면 body가 위로 밀리고 필드·탭바는 고정 → 바닥 안정. field 높이는 relativeTo: .body로 따라 커짐(현행).
· 자동완성은 입력중 키보드 위로 펼침(메신저식) 유지.
