불만 ① 본문이 빽빽하다  ·  불만 ② 미니에선 크고 Pro에선 작게 느껴진다
같은 더미 텍스트로 A(현재) / B(리듬개선) / C(리듬+폭반응 여백) 비교 · 다크모드 토큰 적용

읽는 법

줄간격 차이를 한눈에 보려고 본문을 일부러 길게 넣었습니다(실제 용어는 더 짧음). 폰트 pt는 세 안 모두 동일 — 바뀌는 건 line-height(=SwiftUI lineSpacing), 섹션 간 간격, 라벨↔본문 결속, 그리고 C는 기기 폭별 좌우 여백입니다.
핵심: A는 섹션 라벨이 위/아래 본문과 같은 18 간격이라 "이 라벨이 누구 것인지" 경계가 안 보입니다. 줄간격만 올리는 게 아니라 위계(그룹 경계)를 먼저 잡아야 "빽빽함"이 풀립니다.

각 안의 트레이드오프 · 접근성

A · 현재 — summary lineSpacing 0, 섹션 spacing 통짜 18. 비용: 위계 평평 → 라벨이 떠 보이고 단락이 붙어 답답.

B · 리듬 개선 (추천) — 섹션 사이 26 / 라벨↔본문 9로 비대칭 적용(게슈탈트 근접성), 본문 lineSpacing 6→8, summary에 lineSpacing 부여. 이득: 같은 화면 정보량은 거의 그대로인데 "구획"이 생겨 스캔이 쉬움. 비용: 세로가 약간 길어져 mini에서 스크롤 소폭 증가(허용 범위).

C · B + 폭반응 여백 — B 위에 기기 폭별로 좌우 padding·lineSpacing 미세 조정(폰트 pt는 안 건드림). 이득: "미니에서 커 보임"의 진짜 변수(여백·measure)를 직접 손봄. 비용: 분기 로직 추가 = 유지비. 효과는 미묘해서 B 대비 체감이 작을 수 있음.

기기별 폰트 스케일링은 넣지 않았습니다. iOS 표준은 사용자가 제어하는 Dynamic Type이고, 기기 폭으로 pt를 바꾸면 Dynamic Type 설정과 충돌하거나 접근성이 깨집니다. "미니에서 커 보임"은 폰트가 커서가 아니라 좁은 폭 + 좌우 여백 + 줄간격이 만든 착시 → 해법도 거기에 둡니다.
접근성 공통: 본문은 모두 Font.system(.body) 유지(Dynamic Type 추종). lineSpacing은 pt가 아니라 비율감으로 작동해 큰 글자에서도 안전. textMuted는 섹션 라벨(11~12pt)까지만 — 본문엔 textDim 유지(다크 대비 확보).
